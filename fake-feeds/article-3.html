Trait Objects vs Generics: When Does vtable Overhead Actually Matter?

Carol White · March 10, 2026 · 6 min read

A common piece of advice in the Rust community is to prefer generics over trait objects for performance. This is often repeated without measurement. This article attempts to put actual numbers on the claim.

The Benchmark Setup

We compared two implementations of a simple event handler dispatch loop: one using a Vec<Box<dyn Handler>> and one using monomorphised generics. The loop dispatched 10 million events across eight handler types.

Results

The generic version ran in 42ms. The trait object version ran in 61ms — a 45% overhead. However, when the number of distinct types dropped to two, the vtable version was within 8% of the generic version, likely because the CPU's indirect branch predictor learned the pattern.

When vtable Dispatch Is Fine

If your hot path dispatches to fewer than four distinct concrete types, and those types are called in a consistent pattern, vtable overhead is effectively zero in practice. The branch predictor covers you.

Plugin architectures and event bus implementations — which load handler types at runtime — have no choice but to use trait objects anyway. In these cases the flexibility is worth more than the hypothetical performance gain from generics.

When to Prefer Generics

If you have a tight inner loop dispatching over many distinct types in an unpredictable order — think a scripting language evaluator or a renderer sorting draw calls — generics will win. The compiler can inline aggressively and the CPU avoids mispredictions entirely.

Takeaway

Profile first. The default advice to prefer generics is directionally correct but overstated for most real applications. Use dyn Trait freely for architectural flexibility; switch to generics when a profiler tells you to.
